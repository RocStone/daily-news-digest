## 1. 核心洞察：这到底在解决什么问题

### 问题本质
当前 LLM 有三个「天花板」：
1. **权重静态化**：训练完就冻住了，上下文压缩是有损的，长期记忆不如人类睡一觉后的巩固机制
2. **人类数据耗尽**：Epoch AI 的预测显示，前沿模型消耗的 token 量正逼近互联网公开数据总量，Scaling Law 的燃料快烧完了
3. **算法发现依赖人力**：idea → code → experiment → iterate 这个循环极其昂贵，人类一周只能试几十个想法，AI 能试几万个

### 一句话本质
**用模型自身的能力去突破模型的瓶颈**——让 AI 自己造数据、自己改预训练、自己搜索算法，形成自我改进闭环。

### 为什么有效
不是魔法，核心逻辑很朴素：
- 合成数据有效，是因为互联网文档之间存在 **未被显式利用的跨文档相关性**（Attention is All You Need 论文和 GPT-2 的 PyTorch 实现之间的关联），合成数据把这些隐式关联显式化了
- 自我改进有效，不是因为 AI「更聪明」，而是因为它 **用数量碾压质量**——人类研究员一周跑 13 个实验，AI 跑 30,000 个

## 2. 三条路径的技术解剖

### 路径一：EntiGraph 合成持续训练——给冻住的模型灌新知识

> Setting：**在已有预训练模型上做持续预训练（continued pre-training）**，不是从头训。拿一个现成的 Llama 3 8B base model，在它的权重基础上继续喂数据，让它学会新领域的知识。训练方式依然是 next-token prediction，和预训练一样，只是起点不是随机初始化而是一个已经训练好的 checkpoint。
> **合成数据由外部强模型（GPT-4 级别）生成**，不是 Llama 3 自己生成的。所以路径一包含蒸馏成分——大模型的知识通过合成数据传递给了小模型。但路径一的贡献不在于蒸馏本身（对比组的重写法也是 GPT-4 生成的，效果差很多），而在于 **数据增强策略**：实体组合产生的多样性远超固定 prompt 重写。

> 解决的问题：如何在少量数据上，SFT一个模型。
> 解决方案本质上是数据增强，做法是靠更强的大模型，对少量数据上的大量实体，让强模型描述实体之间的关系，假设样本量只有1k，但是每个样本内的实体量是1k，那么一共就是1m个实体，如果任意取两个实体，让强模型描述他们之间的关系，那就有$C_{1m}^2$，这种描述他们关系的文本就是新的数据，这个数据量不要太多。
> 而实际上用这种数据做SFT效果也巨好。但这本质上也是一种蒸馏。


#### 问题设定：手头有什么，想做什么

**手头的数据**：QuALITY 数据集——265 本专业书籍，总计约 180 万个 token。这些书都是模型预训练时没见过的冷门内容。另外还有约 4000 道对应的高质量四选一多选题，用来测模型对这些书的掌握程度。

> 少量数据想finetuning 怎么办

**想做的事**：闭卷考试。不给模型看书，直接问问题，看它能答对多少。这模拟的是「把知识真正写进权重」而不是临时查阅。

#### 为什么天真的方法全都失败

**方法 A：直接在原始书籍上持续预训练（Naive Continued Pre-training）**
具体做法：拿 Llama 3 base 模型，把 265 本书当训练数据，继续跑 next-token prediction。
结果：准确率从 39% **反而下降**。
原因：180 万 token 太少了。正常预训练需要几十亿甚至上万亿 token 来让 batch size 和 learning rate schedule 工作在合理区间。你拿 180 万 token 去跑，相当于让一个学生只看了两页笔记就去考试，而且这两页还会导致他把之前学的东西忘掉（灾难性遗忘）。除非你做大量的 replay（重放旧数据），否则根本不行。

> 如果样本量少，直接做后训练会当场去世，模型只会变差。

**方法 B：用 LLM 重写源文档（Rewriting）**
具体做法：给 GPT-4 类模型一个固定 prompt 比如「请把这本书像维基百科文章一样重写」，准备了 4 个这样的固定 prompt，反复迭代地应用到每本书上。每次靠 temperature 采样来产生不同的输出。
结果：准确率有一些提升，但斜率不陡，天花板不高。
原因：**多样性耗尽**。prompt 是固定的，唯一的随机性来源是 temperature。当你迭代 200-300 次后，模型输出的内容高度雷同——你本质上是在用稍微不同的措辞说同样的话。这就像让学生抄课本 300 遍，他记住的只是措辞，不是知识结构。

> multi sample 最后多样性是会耗尽的

#### EntiGraph 到底怎么做的：两阶段详解

**第一阶段：实体提取**
输入：一本源文档（比如一本关于 19 世纪蒸汽机发展史的专业书）
操作：给 LLM 一个 prompt：「这是源文档，书中有哪些核心实体能帮助我很好地理解这篇文章？」
输出：一个实体字符串列表。比如：`[James Watt, 冷凝器, Newcomen 蒸汽机, 工业革命, 热效率, 蒸汽压力, 纺织工厂, ...]`

> 提取实体，这里的LLM是GPT4，所以有蒸馏的思想

**第二阶段：实体关系描述（核心步骤）**
操作：从实体列表中 **随机采样一个子集**（比如随机抽 2-3 个实体），然后让 LLM 描述它们之间的关系。
举个具体例子：
- 采样到 `[James Watt, 热效率]` → 模型生成：「James Watt 对 Newcomen 蒸汽机的改进本质上是热效率的革命。他引入独立冷凝器，使得……」
- 采样到 `[冷凝器, 纺织工厂]` → 模型生成：「冷凝器技术的成熟直接推动了纺织工厂的动力革命。在此之前，纺织业依赖水力……」
- 采样到 `[Newcomen 蒸汽机, 蒸汽压力, 工业革命]` → 模型生成：「Newcomen 蒸汽机的根本局限在于大气压力原理，它无法利用高压蒸汽……」

> 让AI描述若干实体之间的关系，一样，也是用GPT4，所以也是蒸馏

**为什么这比重写好得多？**
关键 insight：每次你在 prompt 中 **换一个实体**，这不是微小的扰动——对 Transformer 来说，改变 prompt 中的一个 token 就是改变了输入的词向量，经过多层 attention 后输出会发生 **高度非线性的变化**。所以你不需要靠 temperature 来制造多样性，实体组合本身就是一个组合爆炸级别的多样性来源。如果有 50 个实体，两两组合就有 1225 种，三三组合就有 19600 种。每种组合产出的描述都是从不同角度切入同一批知识。

> 实体组合产生的多样性非常之多，所以比直接采样多次产生的样本量大得多

这就是为什么它叫 **EntiGraph**（Entity Graph）——你把实体当节点，关系描述当边，实际上是在围绕源文档构建一张知识图谱的文本化表示。

**第三阶段：用合成数据做持续预训练**
拿 EntiGraph 生成的所有关系描述文本，加上原始源文档，混合起来对 Llama 3 base 做持续预训练。随着合成 token 数量增加，闭卷准确率持续上升。

> 现在有了大量的样本可以做SFT了

#### 关键结果与 insight

| 方法 | 闭卷准确率 | 备注 |
|---|---|---|
| 随机瞎蒙 | 25% | 四选一基线 |
| Llama 3 base 开箱 | 39% | 模型对冷门书籍一无所知 |
| 直接持续预训练 | <39% | 反而更差 |
| 重写法（最好） | ~45% | 斜率平缓，天花板低 |
| GPT-3.5（静态） | 44% | 参考 |
| GPT-4（静态） | 45% | 参考 |
| **EntiGraph 合成训练** | **56%** | 截距和斜率都远优于重写 |
| Llama 3 开卷（RAG） | 60% | 直接把书塞进 context |
| **EntiGraph + RAG** | **最优** | 两者改进互补 |

**最重要的 insight**：EntiGraph 和 RAG 是互补的，不是替代关系。EntiGraph 把知识写进权重（参数记忆），RAG 在推理时提供上下文（工作记忆）。就像人类既需要长期记忆也需要查资料一样，两者叠加效果最好。这意味着做领域适配时，**不应该二选一，而是两个都做**。

**对你的直接价值**：如果你要给企业做领域模型，操作路径是：提取领域文档的实体 → 生成实体关系描述（放大训练数据 10-100 倍）→ 在基座模型上持续预训练 → 部署时叠加 RAG。这可能是未来开源模型个性化适配的标准范式。

### 路径二：Synthetic Bootstrap Pre-training (SBPT)——自生成数据改善预训练

> Setting：**从头预训练（pre-training from scratch）**，不是在已有模型上微调。整个流程会训练两次：第一次从头训出一个 base model，第二次用真实+合成数据从头再训一个新的 base model。中间有一个 SFT 步骤把第一次的 base model 改造成合成数据生成器，但最终产物是第二次从头训出来的模型。和路径一的本质区别：路径一是在已有模型上「补课」，路径二是证明「如果你用更聪明的方式组织训练数据，从头训出来的模型会更强」。
> **和路径一的另一个关键区别：合成数据由模型自己生成，不引入任何外部更强模型**。路径一用 GPT-4 生成合成数据给 Llama 3 训练，包含蒸馏成分。路径二刻意避免了这一点——论文原文明确说「必须确保看到的是预训练能力的普遍引导提升，而不仅仅是从教师模型中进行的知识蒸馏」。全程只用自己训出来的模型生成数据再训练自己，所以这是 **真正的自我改进**，不是蒸馏。

#### 为什么要碰预训练？从 s1 说起

Zitong Yang 做这个研究的动机来自他参与的 s1 项目。当 OpenAI 发布 o1 时（2024年9-10月），所有人都以为 o1 背后需要海量的人工标注推理数据。但 s1 证明了：**仅 10,000 条思维链数据做 SFT，就能达到 o1-preview 级别**。

这个结果的深层含义是：后训练（SFT、RLHF）不是在「教」模型新能力，而是在 **激活** 预训练已经学到的能力。后训练就像是给一把已经磨好的刀加一个趁手的刀柄——刀本身的锋利程度（预训练质量）才是决定性的。

所以，如果你想看到 **真正的** 自我改进，必须在预训练层面看到提升，而不仅仅是后训练层面。

> 所以 SFT 想起作用，必须是 base model 足够的好。SFT 只是激活而已。

#### 预训练的知识到底从哪来？一个思想实验

假设世界上只有 5 个 token：A、B、C、D、E。训练文本是完全随机生成的字符串（每个位置均匀随机采样一个字符）。用 Transformer 在这些文本上做 next-token prediction 会发生什么？

**什么都学不到**。因为初始化时模型已经给每个 token 分配了 20% 的概率，而数据的真实分布也是均匀的 20%。训练损失从一开始就是最优的，梯度为零。

但自然语言不是随机字符串——token 之间存在 **统计相关性**（统计学视角）或者说 **可压缩的模式**（信息论视角，接近 Shannon 的原始理论）。正是这种结构让 next-token prediction 有东西可学。

**Zitong Yang 的关键观察**：当前预训练范式遗漏了一个巨大的相关性来源——**跨文档的相关性**。

具体例子：
- 《哈利波特》小说 ↔ 三年后的电影剧本：两者描述的是同一个故事，但用完全不同的文本表达
- Attention is All You Need 论文 ↔ GPT-2 的 PyTorch 代码库：论文第 3.2 节写「dot-product attention」，代码第 91 行实现了 `torch.matmul(Q, K.transpose(-2, -1))`——英语描述和 Python 代码之间存在深层关联
- 一篇圣地亚哥咖啡馆指南 ↔ 一篇纽约咖啡文化文章：两者都在讨论咖啡，但从不同城市的视角

当前预训练只是把这些文档按顺序拼成一个长序列去训练，**没有显式地告诉模型这些文档之间有关系**。SBPT 要做的就是把这种跨文档关联利用起来。

> SBPT 利用跨文档相关性的方式和 EntiGraph 完全不同——**不提取实体，用 embedding 相似度**。把每篇文档丢进 Qwen 0.6B embedding 模型得到向量，在向量空间算最近邻。Transformer 论文和 GPT-2 代码的向量天然就近（embedding 模型已经学会语义相似性），不需要知道「为什么相关」，只需要知道「它们相关」。然后 SFT 让模型学会「看到 A 就联想到 B」，生成时这种联想能力把隐式关联显式化为文本。全程不涉及实体提取，也不需要理解代码结构。

#### SBPT 操作流程：五步详解

**Step 1：基础预训练**
数据：DCLM 数据集的一个子集（标准互联网文本）
操作：从头预训练一个 Llama 3 架构的模型（加了 QK LayerNorm 稳定训练），得到一个 checkpoint
输出：一个正常的预训练模型，和你平时见到的 base model 没区别

**Step 2：最近邻向量编码——找到「相关文档对」**
操作：把 DCLM 数据集的每篇文档送入 Qwen 0.6B 的 embedding 模型，得到每篇文档的向量表示。然后在向量空间中计算最近邻。
效果：语义相似的文档会在向量空间中聚在一起。比如 Transformer 论文的向量和它的 PyTorch 实现代码的向量会非常接近。
输出：一张文档关系图，每篇文档都有一组「邻居文档」

**Step 3：合成微调——训练一个「跨文档联想生成器」（最精巧的一步）**
输入：Step 1 的预训练 checkpoint + Step 2 的文档邻居关系
操作：对预训练模型做 SFT。训练目标是：**以一篇文档 d1 为条件（context），最大化其邻居文档 d2 的对数概率（target）**。

具体来说，假设文档 d1 是 Attention is All You Need 论文，它的邻居 d2 可能包括：
- GPT-2 的 PyTorch 实现代码
- 一篇关于 Multi-Head Attention 的博客
- BERT 论文
- 一段 Transformer 的教学视频转录文本
- ……共约 20 个邻居

SFT 的训练样本就是：`context=d1, target=d2_1`、`context=d1, target=d2_2`、……

**为什么这一步让模型进入「高熵状态」？**
先说什么是熵：熵衡量的是一个概率分布的「不确定性」。如果模型看到 d1 后 100% 确定输出某一篇文档，熵为零；如果它在 20 篇文档上各有 5% 的概率，熵就很高。
现在回到 SFT 的训练过程。同一个 d1 出现在 20 条训练样本里，每条的 target 是不同的邻居文档。最大似然估计（MLE）会怎么做？它会找一个概率分布，让这 20 个 target 的联合概率最大。数学上最优解就是：**在这 20 个邻居上近似均匀分配概率**。
但注意——这不是「什么都可能输出」的无差别随机。模型只在 d1 的 **语义邻域内** 高熵：看到 Attention 论文，可能输出 PyTorch 代码、BERT 论文、教学博客中的任何一种，但不会输出一篇烹饪食谱。这是 **结构化的高熵**——有方向的多样性，不是噪声。
这个设计是刻意的：正因为多样性被限制在语义邻域内，生成出来的合成数据才是有意义的跨文档关联，而不是胡说八道。


**Step 4：合成数据生成——用高熵模型「自由联想」**
操作：遍历所有真实文档，对每篇文档用 Step 3 微调后的模型做 temperature=1 的生成。不用任何花哨的 prompt，就是直接让模型接着写。
效果举例：
- 输入一篇圣地亚哥咖啡馆列表 → 输出 A：一篇关于圣地亚哥之旅的游记，中途话题转向浓缩咖啡机的技术细节（源文档完全没提过咖啡机）
- 同一篇输入 → 输出 B：一篇将圣地亚哥咖啡文化与纽约进行对比的文章（「纽约」这个词根本没出现在源文档中）

这些「跑题」的内容不是幻觉——它们是模型利用预训练中学到的 **跨文档关联**，把散落在互联网各处的相关知识「联想」到了一起。咖啡馆列表让模型联想到了咖啡文化，咖啡文化联想到了不同城市的对比，不同城市联想到了旅行体验。这种联想链正是当前预训练没有显式利用的信息。

**Step 5：混合重训——真实数据 + 合成数据从头预训练**
操作：把原始的 DCLM 真实数据和 Step 4 生成的合成数据混在一起，从头训练一个新模型。每当基准方法试图 **重复** 真实数据时（因为数据有限，基准方法只能把同样的数据多跑几遍），SBPT 替换为合成数据。
关键约束：**训练计算量严格匹配**。基准方法和 SBPT 跑的训练 epoch 总数完全相同，唯一区别是 SBPT 用合成数据替代了重复数据。（注意：没有控制生成合成数据的推理计算量，SBPT 总算力更高，但训练算力相同。）

#### 实验设计：三个对比对象

| 对比对象 | 数据访问 | 训练计算量 |
|---|---|---|
| **基准（Baseline）** | 固定数据，不够就重复 | 控制相同 |
| **SBPT** | 固定数据 + 合成数据 | 控制相同 |
| **Oracle** | 无限真实数据 | 控制相同 |

Oracle 是理想上界——如果数据永远不会耗尽，模型能好到什么程度？

#### 关键结果

**训练动态**：
- 早期（训练 token 较少时）：Oracle 和基准表现几乎相同——数据还没重复，没有区别
- 中期：基准开始饱和（曲线变平），因为它在一遍遍重复同样的训练信号，学不到新东西了
- 后期：Oracle 和 SBPT 都继续以 **近似线性** 的速率下降。SBPT 用合成数据有效地「模拟」了拥有无限数据的效果

**定量结果**：

| 设置 | 问答准确率提升（vs 基准） | Oracle 带来的提升 |
|---|---|---|
| 3B 参数 / 200B tokens | 显著 | ~30% |
| 3B 参数 / 1T tokens | 一致 | ~30% |
| 6B 参数 / 1T tokens | 部分指标更高 | ~30% |

三种设置下 Oracle 带来的提升都是约 30%，非常一致。SBPT 能 recover 其中很大一部分。

**合成数据质量随模型规模的变化（最重要的表）**：

| 生成器 | 重复内容比例 | 非事实（幻觉）比例 |
|---|---|---|
| 3B / 200B tokens | 较高 | **~50%** |
| 3B / 1T tokens | 降低 | 显著降低 |
| 6B / 1T tokens | 更低 | 进一步降低 |
| 真实数据 | 最低 | 最低 |

**反直觉发现的深层解读**：
- 从 200B → 1T（同一个 3B 模型，5倍训练量）：事实错误率下降，好理解——模型见了更多数据，知道更多世界知识
- 从 3B → 6B（训练量不变，模型变大）：事实错误率 **也大幅下降**。这才是真正有意思的。训练数据完全相同，环境的总信息熵没变，但更大的模型从相同的数据中提取了更多信息，因此生成时幻觉更少。这暗示：**模型规模本身就是信息提取效率的放大器**，不需要更多数据，只需要更大的「理解力」
- 这意味着如果你把模型做到 70B 甚至更大，合成数据的质量可能接近甚至匹敌真实数据。到那个时候，自我改进闭环就真的闭合了——模型生成的数据好到可以完全替代真实数据

> model 越大，这种跨文档构建出来的数据的质量越高，意味着大模型可能可以自己生成数据，自己训练，直接迭代

### 路径三：AI 设计 AI——让模型自己搜索算法

> Setting：**不训练模型本身，而是搜索更好的训练算法/超参数**。这里有两个角色——「研究员」是一个现成的强 LLM（如 GPT-4 级别），它的权重不变；「被研究的对象」是研究环境中的小模型（nanoGPT 或 GRPO 训练的模型）。研究员通过修改代码来改进被研究对象的训练效果。和路径一二的本质区别：路径一二在改进「数据」，路径三在改进「算法」。

#### 第一性原理：为什么 AI 做 AI 研究在理论上可行？

回到 Karl Popper / Fisher 的科学方法论：科学进步 = **提出假设** + **实验证伪**。

- 提出假设：对 LLM 来说就是写一段文本描述一个想法。LLM 天生就能做这个，只是校准度不够
- 运行实验：在 AI 领域，实验 = 写代码 + 跑 benchmark。SWE-bench 准确率从 20% 飙到 80%，说明 LLM 写代码的能力已经非常强
- AI 领域还有一个独特优势：**极度 benchmark 驱动**。ImageNet、MATH、各种 leaderboard——你有一个明确的数字告诉你做得好不好。这天然适合自动化搜索

#### 研究环境抽象：把 AI 研究形式化为一个 API

Zitong Yang 定义了一个叫 **Research Environment** 的抽象，把 AI 研究变成了一个标准接口：

```
class ResearchEnvironment:
    context: str          # 代码库，定义了研究问题是什么
    sandbox: Resource     # 计算资源（比如 8xA100 或 1xBlackwell）

    def evaluate(code_diff: str) -> float:
        # 1. 把 code_diff 应用到代码库
        # 2. 在 sandbox 中执行 run 命令
        # 3. 运行评估脚本
        # 4. 返回标准输出中的奖励信号（一个数字）
```

**具体实现了两个环境**：

**环境 A：预训练优化（nanoGPT）**
- context：一个独立的 Python 脚本，执行 GPT-2 预训练
- 资源：8 张 A100 GPU
- 评估指标：在固定资源限制下，测试损失达到 3.28 所需的时间（越短越好）
- 排行榜来源：斯坦福 CS336 课程

**环境 B：后训练优化（GRPO 数学推理）**
- context：在 GSM8K 数据集上用 GRPO 算法训练，在 MATH500 上测试
- 资源：1 张 Blackwell GPU（单卡设计是为了让采样器和训练器共享权重，消除多卡通信的工程复杂度）
- 评估指标：MATH 测试准确率（人工验证）
- 来源：也是 CS336 课程作业

#### 自动化研究员的四步循环：详解

**Step 1：构思（Ideation）**
研究员内部有一个「构思器」（Ideator）组件。它接收研究环境的 context（代码库内容），输出一个自然语言描述的想法。
例如输入是 GRPO 训练代码，输出可能是：「可以在训练循环中引入一个上下文缓冲区，在每步推理后存储中间数学事实，供后续步骤使用」

**Step 2：执行（Execution）**
研究员内部的「执行器」接收 context + 想法，输出具体的 code diff。
例如把上面的自然语言想法翻译成：在 GRPO 训练脚本中新增一个 `ContextBuffer` 类，包含 `add_context()` 和 `get_context(query)` 两个方法，在 RL 循环的每个 prompt 处调用。

**Step 3：实验（Experiment）**
把 code diff 送入研究环境的 evaluate 函数。沙盒执行修改后的代码，跑完训练和评估，返回一个数字（比如 MATH 准确率 53%）。

**Step 4：学习（Learn）**
把（想法, code diff, 得分）三元组存入一个「实验记录库」。随着实验积累，这个库越来越大。

#### 演化搜索机制：利用 vs 探索

有了实验记录库后，下一轮构思时，构思器可以执行两种操作：

**利用（Exploit）**：从库中选出得分最高的几个想法，让 LLM 把它们的优势 **组合** 起来生成一个新想法。
例如：想法 A 是「用上下文缓冲区」(+10%)，想法 B 是「调整 learning rate 衰减策略」(+5%)。组合后的想法 C 可能是「用上下文缓冲区 + 余弦退火学习率」。

**探索（Explore）**：要求 LLM 生成一个与库中已有想法 **截然不同** 的全新想法。
例如：库里全是关于训练策略的想法，探索可能产出「修改 tokenizer 的数学符号编码方式」这种完全不同方向的想法。

这个利用-探索的循环一轮轮迭代下去。和遗传算法的核心区别是：这里的「基因」不是数字编码，而是 **自然语言描述的研究想法**，「表达」不是表型，而是 **可执行的 Python 代码**。LLM 同时扮演了交叉（crossover）、变异（mutation）和基因到表型映射（idea → code）三个角色。

#### 关键结果

**后训练任务（GRPO 数学推理）**：
- 起点：48% MATH 准确率
- 搜索后：**69%**
- CS336 课程排行榜人类最好成绩：68%
- 结论：在弱意义上超越了最好的人类学生

**预训练任务（nanoGPT）**：
- 有改进（36分钟 → 优化后更短），但排行榜最好成绩是 2.1 分钟
- 结论：远不如人类。预训练优化的搜索空间太大太没有结构，自动化搜索还不够

#### 一个精彩的定性例子：数学工作记忆模拟

模型在 GRPO 数学推理环境中自主提出了这个想法（不是人类提示的）：

**想法**：创建一个「数学工作记忆模拟」——维护一个 `ContextBuffer` 类，存储数学事实、定义和中间结果。随着模型逐步解题，缓冲区持续更新，为后续推理步骤提供额外上下文。

**代码实现**：
```python
class ContextBuffer:
    def add_context(self, fact: str): ...
    def get_context(self, query: str) -> str: ...

# 在 RL 训练循环中：
buffer = ContextBuffer()
for prompt in batch:
    retrieved = buffer.get_context(prompt)
    augmented_prompt = retrieved + prompt  # 把历史知识附加到当前问题
    # ... 正常的 GRPO 训练 ...
```

**效果**：比基准提升 10%。不是最好的结果，但 Zitong Yang 说这个想法让他非常兴奋——因为他自己就有一本手写笔记本，记录数学技巧（裂项相消、epsilon 球、Jensen 不等式以及何时使用它们）。模型独立地「发明」了和人类数学家同样的工作方法。

**深层 insight**：模型不是在「发现新数学」，而是在发现 **如何更好地组织已有知识来解题**。这和 Meta-Prompting 的核心思想异曲同工——不教模型怎么解题，教它怎么组织解题过程。

#### 串行 > 并行：一个跨领域的规律

| 场景 | 并行方法 | 串行方法 | 结果 |
|---|---|---|---|
| 算法搜索（nanoGPT） | Best-of-N、多数投票 | 演化搜索（每步复用前一步） | 并行几乎无提升，串行持续改进 |
| 数学推理（s1） | 多次独立采样取最优 | 预算强制（追加逗号让模型继续想） | 并行斜率小，串行斜率大 |

**为什么串行更好？** 并行方法（Best-of-N）等价于在搜索空间中独立采样 N 个点取最好的。串行方法等价于从一个点出发，每一步都利用前一步的信息决定下一步往哪走。当搜索空间有层次结构时（绝大多数有意义的问题都是如此），有方向的搜索远比随机撒点高效。

这和 s1 的预算强制技巧本质上是同一个 insight：不是让模型多次独立尝试，而是强迫它在一次尝试中 **持续思考更长时间**。追加一个逗号（强制继续思考）比独立采样 10 次更有价值。

**但两者都会迅速饱和**——无论是搜索还是强制思考，性能都会碰到天花板。这暗示测试时计算量不是万能药，根本能力还是受限于预训练质量（回到了路径二的论点）。

## 3. 定位与对比

### 与 [[2026-03-04 Knuth论文启发与AI科研Skill设计思考]] 的深度对比

Knuth 的「教练模式」和 Zitong Yang 的「AI 设计 AI」在解决同一个问题： **让 AI 做研究** 。但路径截然不同：

| 维度 | Knuth 教练模式 | Zitong Yang 演化搜索 |
|---|---|---|
| 人类角色 | 教练，深度参与每轮迭代 | 设计者，搭好环境后退出 |
| 反馈信号 | 人类判断 + 确定性验证脚本 | 自动化价值函数 |
| 适用范围 | 数学/逻辑（需极速确定性反馈） | 有明确 benchmark 的 AI 任务 |
| 搜索方式 | 人引导的深度优先 | 自动化的利用-探索平衡 |
| 瓶颈 | 人类精力（31 轮就很累了） | 模型构思质量（跑 30,000 个实验但每个想法质量有限） |

**核心分野**：Knuth 依赖人类智慧做「质」，Zitong Yang 依赖自动化做「量」。目前两者都有天花板——串行计算比并行计算更有价值这条规律暗示，纯粹堆量会迅速饱和。

### 与 Prompt 工程体系的关系
新闻 map 中的 prompt 工程文档（Meta-Prompting、DSPy 等）关注的是 **推理时** 如何编排模型能力。Zitong Yang 的工作关注的是 **训练时** 如何提升模型能力本身。两者是不同层面的事情：prompt 工程优化的是「如何更好地使用现有能力」，自我改进优化的是「能力本身」。s1 的教训说明：当预训练能力足够强时，后训练/prompt 工程的边际收益很高；但如果预训练能力不够，再花哨的 prompt 也救不了。

## 4. 直觉建立

### 类比一：EntiGraph 像是「费曼学习法的自动化」
费曼学习法的核心是：用不同的方式反复解释同一个概念，直到真正理解。EntiGraph 做的就是这个——从不同实体视角反复描述同一批知识，每次视角不同，描述就不同，模型就能从多角度「理解」。

### 类比二：SBPT 像是「让学生自己出模拟题」
一个学生学完教材后，自己出模拟题（合成数据），然后做自己出的题。出题过程本身就是对知识的重组和深化，即使题目不完美，做题过程也能巩固理解。模型规模越大，「出的题质量越高」（事实错误越少）。

### 类比三：AI 设计 AI 像是「遗传算法 + 代码解释器」
把每个研究想法看作一个基因，价值函数是适应度。高适应度的想法被组合（crossover），偶尔产生全新变异（exploration）。区别在于「基因」是自然语言描述的想法，「表达」是可执行代码。

## 6. 非显而易见的东西

### 大多数人会忽略的
- **SBPT 的 SFT 阶段让模型进入高熵状态，这不是 bug 而是 feature**。每个 d1 对应 20 个 d2，最大似然估计会让模型输出接近均匀分布。这意味着生成时模型不会锚定在任何特定模式上，天然产生多样性。这个设计非常精巧
- 路径三中， **模型在后训练任务上超过人类、在预训练任务上远不如人类** 。这揭示了一个重要区别：后训练任务（GRPO 数学推理）的搜索空间相对小且结构化，适合自动搜索；预训练优化的搜索空间巨大且缺乏结构，人类的系统性思维依然占优

### 隐含假设
- **假设预训练范式不变**：所有三条路径都建立在「参数化 + 预训练」这个框架上。如果未来出现完全不同的范式（比如持续在线学习、非参数化记忆），这些方法的适用性会大打折扣
- **假设合成数据的分布偏移可以忽略**：SBPT 用模型自己生成的数据训练自己，理论上存在分布坍缩风险。论文通过混合真实数据缓解了这个问题，但没有给出理论保证

### 六个月后最可能被验证/推翻的部分
- **最可能被验证**：EntiGraph 类方法会被工业界广泛采用做 domain adaptation，因为实操成本低且效果好
- **最可能被挑战**：「串行 > 并行」如果被理论化，可能会发现它只在特定条件下成立（比如搜索空间有某种层次结构时），而非普适规律
- **最值得观察**：SBPT 能否在真正大规模（70B+）模型上复现。目前只在 3B-6B 验证，大模型的事实错误率下降趋势如果持续，意味着足够大的模型可以实现真正的自我改进闭环

## 7. 与爱因斯坦类比的批判性审视

答辩结语中爱因斯坦场方程的类比很有感染力，但逻辑上有漏洞：
- 场方程预言宇宙膨胀，是因为方程编码了 **物理规律的数学结构**，这个结构比任何人的直觉都更忠实于现实
- AI 系统编码的是 **训练数据的统计结构**，不是物理规律。模型「超越创造者」的方式目前还是量的优势，不是质的突破
- 更精确的类比应该是： **AI 就像一台极其强大的统计显微镜** ——它能看到人类看不到的数据中的模式，但它看到的东西受限于数据本身，而非像场方程那样指向数据之外的真理

这不是说 AI 不可能超越人类，而是说目前的证据还不足以支撑「必然性」这个结论。正确的态度是： **可能性很高，但路径还不清楚。**

## 9. 与 Karpathy autoresearch 的深度对比：路径三的工程化落地

> 2026年3月8日，Andrej Karpathy 开源了 [autoresearch](https://github.com/karpathy/autoresearch)。核心思路：给 AI agent 一个完整的 GPT 训练脚本，让它自主修改代码、训练5分钟、评估结果、保留或丢弃，循环往复。一觉醒来收获约100个实验结果和一个更好的模型。

### 9.1 autoresearch 工作原理

**三个文件构成整个系统**：
- **prepare.py** ：不可修改。处理数据下载、BPE tokenizer 训练、dataloader 和评估函数。相当于「实验室的基础设施」
- **train.py** ：agent 唯一允许修改的文件，约630行。包含完整的 GPT 模型定义、Muon+AdamW 优化器、训练循环。agent 可以改架构、改超参、改 batch size、改优化策略——随便改，只要改这一个文件
- **program.md** ：人类写的 Markdown 指令，告诉 agent 研究方向和约束。Karpathy 说这个文件已经「90% 由 AI 撰写」

**实验循环**：
1. Agent 读取 program.md 和 train.py，提出一个修改想法
2. 直接修改 train.py 代码
3. 在单张 GPU 上训练恰好 5 分钟（wall clock，不含启动和编译时间）
4. 读取 val_bpb（validation bits per byte，越低越好）
5. 如果 val_bpb 改善 → git commit，保留修改，在此基础上继续（advance branch）
6. 如果 val_bpb 不变或变差 → git reset，回退到上一个好的状态
7. 重复

**实测数据**：一次典型运行跑了 83 个实验，其中 15 个被保留为改进。agent 在 git feature branch 上逐步积累提交。

### 9.2 核心对比：路径三 vs autoresearch

| 维度 | Zitong Yang 路径三 | Karpathy autoresearch |
|---|---|---|
| 定位 | 学术研究框架，博士论文 | 工程实践工具，开源项目 |
| 搜索策略 | **演化搜索**：维护想法库，exploit（组合最优想法）+ explore（生成全新方向） | **贪心爬山**：改善就保留，不改善就回退，在当前最优点上继续 |
| 想法生成 | 显式两步：先用自然语言描述想法（Ideator），再翻译为代码（Executor） | 隐式一步：agent 直接修改代码，想法和执行融为一体 |
| 实验记录 | 存储（想法, code_diff, 得分）三元组，后续构思时参考历史 | git log 作为实验历史，但主要基于当前最优状态继续 |
| 时间约束 | 无固定时间限制，nanoGPT 环境测的是「达到目标 loss 所需时间」 | **固定5分钟**，无论改了什么都只训5分钟，用同一把尺量所有实验 |
| 算力规模 | 8xA100（预训练）/ 1xBlackwell（后训练），跑 5,000-30,000 个实验 | **单张 GPU**，约100个实验/夜 |
| 研究范围 | 预训练优化 + 后训练优化（GRPO 数学推理）两个环境 | 仅预训练优化 |
| 受众 | 有大量算力的研究机构 | **个人研究者/爱好者**，一张卡就能跑 |
| 形式化程度 | 定义了 ResearchEnvironment 抽象类，有完整的 API 设计 | 三个文件，零框架，极简主义 |
| 人类参与 | 搭好环境后完全退出 | 人类通过 program.md 持续引导研究方向 |

### 9.3 本质一致性：同一个 idea 的两种表达

**底层逻辑完全相同**。两者共享的核心 insight 是：

> 把 AI 研究形式化为 **代码修改 → 训练 → 评估 → 反馈** 的闭环，然后让 LLM 自动执行这个循环。

具体来说：
1. **研究环境 = 可执行的代码 + 明确的评估指标**。Yang 写了 ResearchEnvironment 类，Karpathy 用 prepare.py + val_bpb 实现了同样的东西——只是没抽象成类
2. **AI 研究员 = 能读代码、改代码、理解实验结果的 LLM**。Yang 用 GPT-4 级别模型做 Ideator+Executor，Karpathy 用 Claude/GPT-4 做 agent——角色完全一样
3. **串行迭代 > 独立采样**。Yang 的演化搜索和 Karpathy 的贪心爬山都是串行的：每一步基于前一步的结果。这与论文中「串行 > 并行」的结论完美吻合
4. **搜索空间相同**：都是在模型架构、超参数、优化器策略这个空间里搜索

所以回答问题：**是的，autoresearch 和路径三在核心思想上完全一致**。Karpathy 做的就是 Yang 论文中 nanoGPT 预训练环境的实战版本。

### 9.4 关键差异：工程哲学的分野

虽然核心思想一致，但 **实现哲学截然不同**，这些差异本身比相似性更有启发：

**差异一：搜索策略的简化——贪心 vs 演化**

Yang 的演化搜索维护一个「种群」，高分想法被组合交叉产生后代，同时保留探索分支。Karpathy 直接砍掉了这一切——只保留当前最优，改善就前进，不改善就回退。

为什么 Karpathy 敢这么简化？因为他的约束不同：
- 单 GPU、一夜100个实验，不需要高效搜索——暴力试就行
- agent 每次都能看到完整的 train.py（只有630行），有充分的上下文理解当前状态
- program.md 由人类（或半人类半AI）撰写，提供了隐式的搜索方向引导

而 Yang 的设置中，30,000 个实验如果用纯贪心，很容易陷入局部最优。演化搜索的 explore 分支是为了跳出局部最优设计的。**规模越大，搜索策略越重要**。

**差异二：可达性的极端对比**

Yang 的路径三需要 8xA100 跑数万实验 → 数万到十几万 GPU 小时 → 只有 Thinking Machine Lab 和斯坦福级别的机构能做。

Karpathy 只需要一张消费级 GPU → 一夜电费 → 任何有 GPU 的人都能跑。

这不只是工程妥协，而是一个 **哲学选择**：Karpathy 相信 autoresearch 的价值不在于找到全局最优，而在于**让每个研究者都能体验 AI 做 AI 研究的工作流**。就像他做 nanoGPT 不是为了训出最好的模型，而是为了让所有人都能理解 GPT 训练。

**差异三：人类角色的光谱**

| | Yang 路径三 | Karpathy autoresearch |
|---|---|---|
| 设计阶段 | 人类定义环境和评估指标 | 人类写 program.md 定义研究方向 |
| 执行阶段 | AI 完全自主 | AI 自主，但 program.md 可随时更新引导 |
| 反馈阶段 | 纯自动化价值函数 | 纯自动化（val_bpb） |

Karpathy 的 program.md 是一个精妙的设计：它让人类可以在不碰代码的情况下引导 AI 的研究方向，类似于给研究生写一份「研究指导意见」。Yang 的框架里没有这个中间层——一旦环境搭好，人类就完全退场了。

program.md 的存在意味着 autoresearch 实际上是 **人机协作的**，不是纯自主的。但 Karpathy 说这个文件已经「90%由AI撰写」——暗示未来人类连这最后10%的参与也可能被替代。

### 9.5 真知灼见：从对比中提炼的深层 insight

**Insight 1：好的研究基础设施比好的搜索算法更重要**

Yang 花大量篇幅设计演化搜索策略（exploit/explore、想法库、组合交叉）。Karpathy 用最简单的贪心爬山就拿到了不错的结果。这暗示：**真正的杠杆点不在搜索策略，而在实验环境的设计**——5分钟固定预算、单一评估指标、单文件修改范围、完整代码在 context window 内。当环境足够好时，即使是最简单的搜索策略也能工作。

这和 Bitter Lesson 的精神一致：不要在巧妙的搜索策略上过度投入，把精力花在让系统能 scale 的基础设施上。

**Insight 2：固定时间预算是一个被低估的设计决策**

Yang 用「达到目标 loss 的时间」作为评估指标，Karpathy 用「固定5分钟内的 val_bpb」。看似小差别，实际影响深远：
- 固定时间让 **所有实验自动可比**，不需要额外的归一化或对齐
- 固定时间 **天然惩罚过慢的架构**——如果你的模型太大，5分钟内训不出好结果，就会被淘汰。这隐式地把「效率」编码进了评估指标
- 固定时间让 **实验调度变得确定性**——你精确知道一夜能跑多少个实验

Yang 的「达到目标 loss 的时间」更灵活但也更复杂——不同实验跑的时间不同，调度变得不确定，实验之间的可比性也需要更多控制。

**Insight 3：autoresearch 验证了路径三的「后训练优于预训练」结论**

Yang 的发现：AI 在后训练优化（GRPO 数学推理）上超越人类，但在预训练优化（nanoGPT）上远不如人类。Karpathy 的 autoresearch 目前只做预训练优化，而且从社区反馈看，改进幅度有限（83个实验只保留15个）。

这两个独立数据点指向同一个结论：**预训练优化的搜索空间太大太无结构，当前的 AI 研究员还不够**。如果 Karpathy 把 autoresearch 扩展到后训练优化（比如 RLHF/GRPO 调参），效果很可能会好得多——因为那个搜索空间更小、更结构化、反馈信号更明确。

**Insight 4：从学术论文到开源工具的「降维打击」速度在加快**

Yang 的博士答辩是 2026 年 3 月初，Karpathy 的 autoresearch 开源是 2026 年 3 月 8 日。间隔 **不到一周**。当然 Karpathy 的工作可能独立于 Yang（核心思路并不新），但这个时间巧合说明：**学术界刚把一个想法系统化论证，工程界几乎同时就拿出了可用的工具**。

更值得注意的是：Yang 的论文需要斯坦福+Thinking Machine Lab 的资源才能做实验，Karpathy 的工具让任何有一张 GPU 的人都能做类似的事。**学术研究的门槛正在被工程开源以极快的速度拉低**。

**Insight 5：program.md 是「AI 研究的 Prompt Engineering」**

Karpathy 说「你不再直接写 Python 代码做研究了，你在写 Markdown 文件来编程 AI 做研究」。这和路径三中 Yang 设计 ResearchEnvironment 的 context 字符串本质相同——都是在 **用自然语言定义 AI 研究员的工作范围**。

这暗示一个趋势：未来 AI 研究的核心技能可能从「写代码跑实验」转向「设计实验环境和指令」。就像 DevOps 把运维从手工操作变成写 Infrastructure as Code，AI 研究可能从手工实验变成写 Research as Markdown。

### 9.6 总结：同源异流

Zitong Yang 的路径三和 Karpathy 的 autoresearch 是 **同一个核心思想的学术版和工程版**：

- **学术版** 追求完备性：形式化抽象、演化搜索、大规模实验、统计显著性、与人类基线对比
- **工程版** 追求可用性：极简设计、贪心搜索、单 GPU、一夜可跑、开源可复现

两者结合来看，「让 AI 自主做 AI 研究」这件事已经从博士论文的概念验证阶段进入了 **任何人都可以上手的工具阶段**。核心 idea 不再有门槛，**真正的竞争将转向**：谁能设计出更好的实验环境（evaluation infrastructure）、更聪明的搜索策略（从贪心到演化到更高级的方法）、以及更大规模的算力调度。
